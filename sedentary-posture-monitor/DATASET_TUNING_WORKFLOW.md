# 数据集验证与算法调准流程

目标：
- 使用 `dataset/data.csv` 验证当前体态识别算法。
- 在尽量不改变现有产品功能的前提下，提高分类正确率。
- 保持久坐提醒、UI 展示、AI 报告、日志流程基本不变，只修改姿态判定层。

---

## 1. 当前问题定义

现在的系统已经能完成：
- 摄像头实时检测
- MediaPipe 关键点提取
- 体态判断
- 久坐提醒
- 数据看板展示
- AI 报告生成

但在真实测试中，算法存在两个明显问题：

1. 大量样本会被判成“离屏过近”或“不端正”。
2. `TUP / TLF / TLB / TLR / TLL` 的区分不够稳定，尤其是前倾、后仰、端正之间容易混淆。

因此，本轮改进重点不是重做功能，而是：
- 用数据集验证当前算法
- 根据验证结果调整阈值和特征组合
- 最小化影响现有功能

---

## 2. 数据集情况

数据文件：
- `dataset/data.csv`

数据特征：
- 每行一个样本
- 33 维关键点坐标
- 其中上半身 11 个核心关节可用于当前算法验证
- 标签包括：
  - `TUP`：端正坐姿
  - `TLB`：身体后仰 / 瘫坐
  - `TLF`：身体前倾 / 探颈
  - `TLR`：身体向右歪斜
  - `TLL`：身体向左歪斜

数据分布：
- `TLF`: 1897
- `TUP`: 1615
- `TLB`: 442
- `TLL`: 420
- `TLR`: 420

说明：
- 数据是可用于验证的。
- 但它并不是和当前算法完全一致的标签体系。
- 当前算法更偏向“正面摄像头的久坐姿态监测”，所以最适合验证：
  - 前倾
  - 后仰
  - 左右歪斜
  - 正常端正

## 4. 目前算法机制解释

当前系统不是端到端分类器，而是一个“关键点检测 + 几何特征计算 + 规则判定 + 状态机防抖”的管线。

### 4.1 整体流程

1. `MediaPipe Pose` 从摄像头帧中提取 33 个关键点。
2. `analyzePose()` 计算几何特征。
3. `deriveStatus()` 把连续姿态特征映射成离散类别。
4. `PostureStateMachine` 再对连续帧做迟滞过滤，避免单帧误报。
5. 前端再把结果用于骨架渲染、告警、久坐提醒和日志上报。

因此，当前系统的核心不是“神经网络端到端分类”，而是“可解释的规则判定”。

### 4.2 使用的关键点

当前主要使用这些点：
- `nose`
- `left_eye`
- `right_eye`
- `left_shoulder`
- `right_shoulder`
- `left_hip`
- `right_hip`

这些点足够支撑当前任务，因为你要判断的是：
- 头颈是否前伸
- 肩是否不平
- 躯干是否侧歪
- 身体是否前倾或后仰

### 4.3 颈部前倾角 `neckAngle`

做法：
- 先取双肩中点
- 再看鼻子相对于双肩中点的位置
- 鼻子越偏离双肩中点的垂直方向，`neckAngle` 越大
- 鼻子 `z` 值比肩更靠前时，再叠加一个深度惩罚
- 最后减去个性化基线，避免不同身高/坐姿习惯导致的偏差

作用：
- 主要识别头颈前伸、探头、驼背前倾

### 4.4 高低肩 `shoulderDiff`

做法：
- 用左右肩的 `y` 差来表示高低肩
- `y` 差越大，肩越不平
- 再转换成像素尺度，便于前端显示

作用：
- 识别单侧耸肩、肩部不平衡

### 4.5 离屏距离 `screenDistance`

做法：
- 用双肩归一化宽度做反比估计
- 肩宽越小，通常说明人离摄像头越远
- 肩宽越大，通常说明人离摄像头越近

作用：
- 识别“离屏过近”
- 但这个特征如果阈值过宽，会和前倾、后仰混在一起

### 4.6 躯干侧倾 `torsoTilt`

做法：
- 取肩中点和髋中点的连线，与垂直方向求角
- 如果肩中点和髋中点明显错位，就说明躯干在左右歪

作用：
- 识别身体向左/向右歪斜

### 4.7 头侧倾 `headTilt`

做法：
- 通过左右眼连线与水平线夹角计算

作用：
- 辅助判断低头、歪头、姿态失衡

### 4.8 专注度 `gazeFocus`

`gazeFocus` 不是“真正的注意力”，而是一个姿态/视线综合分：
- 鼻子是否居中
- 双眼是否都可见
- 头是否明显歪
- 颈是否明显前倾

作用：
- 给看板和久坐提醒做辅助状态输出

### 4.9 新增深度差 `torsoDepthDelta`

这是这轮调参最关键的新增特征：
- 用肩中点和髋中点的 `z` 差值来区分前后倾
- `torsoDepthDelta < 0` 更偏前倾
- `torsoDepthDelta > 0` 更偏后仰

为什么要加这个：
- 原算法对 `TUP / TLF / TLB` 区分不够
- 仅靠 `neckAngle` 很容易把端正和前倾混掉
- 数据集里前后倾的差异更多体现在深度，而不只是二维平面角度

### 4.10 最终判定逻辑 `deriveStatus()`

当前最终分类逻辑是：
- `TLL / TLR` 先看 `torsoTilt`
- `TLB` 再看 `torsoDepthDelta > 0.07`
- `TLF` 再看 `torsoDepthDelta < -0.08` 或 `neckAngle > 24`
- 否则归为 `TUP`

这套规则的特点是：
- 保留原有的判定结构
- 只把前后倾的区分增强
- 尽量不碰久坐提醒和其他功能

## 5. 基线分析结果

### 5.1 类别可分性

初步观察到：
- `TLR / TLL` 主要由躯干侧倾区分，分离度较高。
- `TLF / TLB` 主要由肩-髋深度差区分。
- `TUP` 与 `TLF / TLB` 容易混淆。

### 5.2 基线启发式准确率

用一版简单启发式规则做基线测试，得到约：
- `85.11%`

主要误差来源：
- `TUP -> TLF`
- `TUP -> TLB`
- 少量 `TLF -> TLL / TUP`

说明：
- 当前算法方向是对的。
- 但“前后倾”和“离屏过近”的边界还不够细。

### 5.3 实际离线评估结果

根据 `dataset/evaluation_report.md` 的网格搜索结果：
- 基线规则准确率: `85.11%`
- 第一轮最佳规则准确率: `90.22%`
- 第一轮最佳阈值:
  - `side_thresh = 14`
  - `back_thresh = 0.08`
  - `front_thresh = -0.08`
  - `neck_thresh = 22`

主要错分模式：
- `TUP -> TLF`
- `TLF -> TLL`
- `TLF -> TUP`
- `TLB -> TUP`

这说明：
- `TUP` 和 `TLF` 依旧是主要边界
- `TLR / TLL` 已经相对稳定
- 因此后续优化应主要围绕“端正 vs 前倾/后仰”的区分展开

### 5.4 新一轮最小改动搜索结果

在不改变久坐提醒和主流程的前提下，进一步尝试更保守的阈值后，离线测试可提升到约：
- `92.12%`

对应阈值组合：
- `side_thresh = 15`
- `back_thresh = 0.07`
- `front_thresh = -0.08`
- `neck_thresh = 24`

这组结果说明：
- 稍微放宽颈角阈值后，`TUP -> TLF` 的误判会下降。
- 稍微收紧后仰阈值后，`TLB` 的判定会更稳定。
- 当前最值得保留的是“只动阈值，不动功能链路”的策略。

## 6. 最小改动调参思路

本轮只改“体态判定层”，不动其他功能。

### 6.1 保持不变的模块

以下功能尽量保持原样：
- 摄像头开关
- 久坐提醒
- 数据看板
- AI 报告
- 日志存储
- 导出功能
- 页面布局和交互

### 6.2 只改动算法核心

只改：
- `src/lib/postureAnalyzer.ts`
- `src/types.ts`
- `src/components/CameraWorkspace.tsx`

核心改动：
- 新增 `torsoDepthDelta`
- 用肩-髋深度差区分前倾 / 后仰
- 收紧 `isTooClose` 的触发条件
- 保留 `neckAngle`、`torsoTilt`、`shoulderDiff` 的原逻辑

### 6.3 为什么这样改

因为这类问题的本质不是“所有东西都错了”，而是：
- 现有规则太粗
- 前后倾和距离感混在一起
- 需要把“姿态类型”拆得更清楚

所以最优方案不是重写系统，而是增强判定特征。

对应效果：
- `TUP` 与 `TLF / TLB` 的区分更清楚
- `TLB` 不再主要依赖 `screenDistance`
- `TLR / TLL` 继续由 `torsoTilt` 保持稳定区分

### 7.2 `src/types.ts`

- 增加 `torsoDepthDelta`
- 增加 `isForwardLeaning`
- 增加 `isBackwardLeaning`

对应效果：
- 上层日志和前端可以看到新的判定信号
- 便于后续继续分析误判样本

### 7.3 `src/components/CameraWorkspace.tsx`

改动内容：
- 将 `torsoDepthDelta` 透传到实时日志
- 将 `isForwardLeaning / isBackwardLeaning` 透传到前端状态

对应效果：
- 前端和后端日志字段与算法保持一致
- 实时场景和离线评估可以统一排查

### 7.4 阈值修改记录

第一轮离线搜索后，代码采用的阈值为：
- `side_thresh = 14`
- `back_thresh = 0.08`
- `front_thresh = -0.08`
- `neck_thresh = 22`

第二轮最小改动搜索后，进一步收紧为：
- `side_thresh = 15`
- `back_thresh = 0.07`
- `front_thresh = -0.08`
- `neck_thresh = 24`

### 7.5 对应测试指标变化

离线评估结果：
- 基线规则准确率: `85.11%`
- 第一轮最佳规则准确率: `90.22%`
- 第二轮最小改动搜索后: `92.12%`

主要改善点：
- `TUP -> TLF` 的误判下降
- `TLB` 的稳定性提高
- `TLR / TLL` 保持高可分性

主要残余误差：
- `TUP -> TLF` 仍是最多的错分
- 少量 `TLF -> TUP`
- 少量 `TLF -> TLL`

说明：
- 这套数据集已经能证明当前系统的规则判定是可行的
- 如果继续提升，还需要增加更精细的前后倾判别特征，但这会开始接近更大幅度的算法改造

### 7.6 实时摄像头误报修正说明

在真实摄像头场景中，如果发现“几乎所有姿势都提示颈椎前倾”，说明单独使用 `neckAngle` 的判定过敏了。

因此后续在实时场景中做了更保守的联合判断：
- `neckAngle` 不再单独决定前倾
- 需要同时满足 `torsoDepthDelta` 的前倾方向
- `screenDistance` 只作为辅助条件，不再直接主导前倾判定

这类修改的目的不是提高离线刷分，而是降低实时场景误报率，让普通端正坐姿不至于频繁触发“颈椎前倾”。

### 7.7 进一步收紧的实时策略

如果实时场景仍然存在“整体过敏”的问题，说明需要继续提高判定门槛：
- 把前倾门槛从“轻微偏负”提高到“明显偏负”
- 把前倾颈角门槛从 `12` 进一步提高
- 把 `isTooClose` 的触发条件再收紧

这样做会略微牺牲灵敏度，但可以显著减少“正常坐姿也报警”的问题。

### 7.8 启动保护策略

如果出现“刚打开摄像头，端正坐姿就立刻报前倾”，说明系统启动阶段还没有稳定下来。

因此增加了启动保护：
- 摄像头打开后的前几秒先观测，不触发前倾/后仰/高低肩告警
- 让用户先完成入镜和坐姿稳定，再进入正式判定
- 默认基线也调得更保守，避免未校准时把正常姿态判得过严

这类保护属于“减少启动误报”的工程处理，不会改变最终的姿态分类逻辑。

### 7.9 正式判定再收紧

如果出现“前 8 秒正常，后面持续报前倾”，说明问题不在启动，而在正式判定阈值仍然偏松。

因此正式模式进一步收紧为：
- 前倾必须同时满足“深度明显偏负”和“颈角明显偏大”
- 取消“颈角高但深度接近中性也算前倾”的宽松分支
- 让 `isSlouched` 更偏向真正的弯腰/探颈，而不是普通坐姿的轻微头位偏差

这一步的目标是先把正常坐姿保住，再慢慢提高灵敏度，而不是一开始就追求高召回。

### 7.10 调试面板

为了定位“到底是哪一个指标把正常坐姿推成前倾”，在摄像头工作台里增加了一个实时调试面板，展示：
- `neckAngle`
- `torsoDepthDelta`
- `screenDistance`
- `postureStatus`
- `isForwardLeaning`
- `isBackwardLeaning`
- `isTorsoTilted`
- `confidence`

这个面板只用于调试，不影响正式功能。它的作用是帮助快速定位：
- 是颈角过敏
- 还是深度差过敏
- 还是距离阈值过严
- 或者是校准基线没有生效

### 7.11 顶部状态文案修正

之前顶部状态栏会把 `isSlouched` 直接显示成“颈椎前倾”，这会造成误导。

现在改成按 `postureStatus` 显示：
- `good` 显示“端正”
- `warning` 显示“姿态异常”
- `danger` 显示“严重异常”

这样可以避免把“只是阈值触发的 warning”误读成“已经明确前倾”。

## 8. 推荐的验证流程

### Step 1: 数据集离线验证

用 `dataset/data.csv` 做离线验证，查看每类样本的预测结果。

重点看：
- `TUP` 是否被误判为前倾/后仰
- `TLF` 是否能稳定识别
- `TLB` 是否会被误判为离屏过近
- `TLR / TLL` 是否可以靠躯干侧倾明显区分

### Step 2: 统计混淆情况

建议输出：
- 总准确率
- 每类召回率
- 混淆矩阵

重点关注：
- `TUP -> TLF`
- `TUP -> TLB`
- `TLF -> TUP`
- `TLF -> TLL`

### Step 3: 调阈值

建议优先调这些参数：
- `torsoDepthDelta` 的前倾/后仰阈值
- `neckAngle` 的前倾阈值
- `torsoTilt` 的左右歪斜阈值
- `isTooClose` 的触发条件

### Step 4: 回归实时摄像头

修改后必须回到实时场景检查：
- 识别是否更稳定
- 久坐提醒是否还正常
- 是否出现误报增多

## 9. 预期结果

如果阈值调得合适，预期会有这些变化：
- `TUP` 不再频繁掉到 `TLF / TLB`
- `TLF` 与 `TLB` 能更稳定分开
- `TLR / TLL` 仍由躯干侧倾判定
- 久坐提醒功能保持不变
- UI 和业务流程保持不变

## 10. 后续工作建议

1. 先用数据集完成一次离线评估。
2. 再根据混淆矩阵微调阈值。
3. 最后用真实摄像头回归测试。
4. 如果有必要，再把评估脚本单独固化成工具。
